## 原始构想

一个 **“基于Tauri的本地优先、插件化通用对象管理框架”**。

一个item管理项目，然后，按条目记录，条目有种类，每一类有自己的属性（类似zotero的journal，book等），条目可以按group分组和标签分类，group可以多级，标签只有一级。条目下可有附件。一个条目可以多个group或者标签。每个条目对应一个哈希码，有一个sqlite记录哈希码，条目名，group和标签信息，可以自定义条目种类。

这个程序自身只提供就是基本的类wiki——之前说的item，三栏布局，左侧group树和标签、右侧条目信息与附件目录文件树（条目对应一个当前仓库目录下条目哈希码为名的一个文件夹，下面可以随意放目录、文件），中间是条目列表。条目目录下默认创建一个与条目同名的markdown，用于展示详细信息。然后给各个接口提供的LLM控制能力。 然后是插件，一个是中间面板的样式美化（也可以默认提供gallery这类基础面板，插件拓展的就是日历面板啊、藏书/影音/游戏库的强化gallery与信息展示面板啊、统计信息面板啊），一个是侧边栏的强化，一些大的功能插件都可以这加一个页面进行。例如一个跟zotero联动的文献管理在侧边栏展示best bib生成的bib解析后的引文列表，还比如这里也可以放统计信息。最后是一些批量操作类和AI强化类的、skill啊、prompt注入啊、与zotero里面插件联动自带生成条目啊等等。  如此一来自己拼接插件或者提供默认组合就能构成日程管理、个人wiki、资源管理、记账等等专用管理仓库

## 功能清单

以下是一份结构化的**功能需求清单（FRS）**，分为核心引擎、界面布局、文件体系、插件系统、AI集成和工作区六大板块：

***

### 1. 核心数据引擎（Rust 后端 · 原子级操作）

*所有业务逻辑均在此层完成，前端仅通过IPC调用，不得直连数据库。*

- **通用对象（Item）模型**：所有条目统一用**12字符纯随机哈希码**作为主键（`sha256(random_16_bytes)` 截前 12 位），包含创建/修改时间戳。

- **动态类型系统（Type System）**：支持用户自定义”条目种类”（如书籍、任务、联系人）。每种类型可动态配置其专属属性集（字段名、数据类型）。Phase 1 支持 `text`（文本）和 `checkbox`（勾选框）两种字段类型；后续扩展数字、日期、关联等类型。

- **多级分组（Group）**：支持无限层级的树状目录，用于条目的物理或逻辑归类。

- **一级标签（Tag）**：支持给条目打上多个扁平标签，用于跨分组的快速筛选。

- **关联关系（Relation）**（Phase 2）：支持条目间的双向链接。条目 Markdown 中使用**标准 Markdown 链接语法** `[文本](../哈希文件夹/)` 指向目标条目文件夹——无需 `[[]]` 等非标准语法，任何编辑器均可正确渲染。程序解析这些标准链接并存入 `item_links` 表，自动生成反向链接（”谁链接了我”），双向互指时自动消除重复显示。条目属性字段中的链接使用程序内部链接机制，由 UI 层面负责跳转。

- **通用查询引擎**：提供统一的过滤/排序接口，支持按类型、标签、分组、自定义属性值及日期范围组合查询。

### 2. 基础界面布局（前端宿主 · 固定骨架）

*主程序提供固定的UI容器，内容随插件或工作区切换而变化。*

- **左侧面板（导航与筛选）**：

  - **分组树**：展示多级分组，支持拖拽移动条目至不同分组。

  - **标签云/列表**：展示所有一级标签，点击筛选当前视图。

- **中间面板（动态内容区）**：

  - **默认列表视图**：以表格形式展示条目，表头根据当前选中的“类型”动态渲染属性列。

  - **视图插槽**：提供容器用于挂载第三方视图插件（如日历、画廊、看板）。

- **右侧面板（详情与资产）**：

  - **属性编辑区**：根据条目的“类型”动态生成表单，用于查看和编辑自定义属性。

  - **附件目录树**：展示该条目对应仓库文件夹下的所有子文件和子目录（树状结构）。

### 3. 文件与附件管理体系（哈希存储）

*分离数据与文件，保证数据整洁与文件自由。*

- **仓库（Repository）机制**：应用可打开/切换不同的本地仓库目录。每个仓库根目录下包含 `.vault/`（数据库 + 应用状态）和以条目哈希码命名的文件夹。

- **哈希文件夹**：每个条目在仓库根目录下拥有一个以自身 12 字符哈希码命名的独立文件夹。

- **自由存放**：用户可在该文件夹内随意增删改任何文件及子目录，应用自动识别并在右侧“附件目录树”中展示。

- **默认备注文件**：条目创建时，自动在哈希文件夹下生成 `<条目名>.md` 文件（内容 `# 条目名`），用于富文本详细描述。双击文件用系统默认程序打开——App 不做内建编辑器。

### 4. 插件扩展系统（前端生态 · 动态加载）

*所有插件均为纯前端（TS/JS）代码，无需重新编译主程序，下载即用。*

- **插件清单（Manifest）**：定义插件名称、版本、入口文件及**扩展点（Contribution Points）**。

- **支持的四类扩展点**：

  1. **中间视图替换**：接管中间面板，实现日历、画廊、甘特图等专属渲染。
  2. **侧边栏/底部面板注入**：在主界面左右侧或底部添加自定义工具栏或信息面板（如Zotero联动列表、全局统计仪表盘）。
  3. **右键菜单/命令面板动作**：注册全局操作，选中条目后可触发批量修改、AI分析、导出等。
  4. **生命周期钩子（Hooks）**：监听“条目创建”、“属性更新”等事件，用于执行自动化脚本。

### 5. AI 与智能体（ACP）集成

*利用AI增强数据管理，而非将其作为附庸。*

- **原生AI网关（Rust层）**：后端封装对本地（Ollama）或云端（OpenAI）API的HTTPS调用，前端插件无需处理复杂的网络与鉴权。

- **结构化上下文注入**：传递给LLM的数据不是纯文本，而是当前条目或筛选结果的标准JSON结构体，便于模型精确处理（非RAG模糊检索）。

- **智能插件示例**：利用上述接口，可开发“自动打标签”、“根据标题生成Markdown摘要”、“智能周报生成”等Skill插件。

### 6. 工作区与场景组合（Configuration as Code）

*数据不动，视图配置动，一套数据支撑无数种管理场景。*

- **工作区快照（Workspace Snapshot）**：保存当前界面布局的“配置快照”。快照记录内容包括：左侧显示的特定分组/标签筛选、中间调用的视图插件、右侧展示的属性模板。

- **场景组合切换**：用户无需更换仓库，只需切换工作区模式即可实现应用形态的剧变。

  - *切换“文献管理”* → 显示BibTex详情面板 + 中间Zotero风格画廊。

  - *切换“日程规划”* → 过滤仅显示带日期属性的条目 + 中间日历插件。

  - *切换“资产盘点”* → 显示带金额/购买日期的表格 + 右侧文件树预览附件发票。

***

### 💡 补充的架构约束（技术选型落地点）

- **后端（Rust）**：仅依赖 `tauri`、`sqlx`/`rusqlite`、`serde`、`tokio`。**不引入**任何与UI渲染或前端框架相关的依赖。

- **前端（TypeScript）**：采用 **Vue 3** 构建宿主和插件容器。插件编译为标准 Web Components（`defineCustomElement`），实现框架无关分发——第三方插件无需依赖 Vue 即可加载。插件必须导出符合 `PluginModule` 接口（`mount`、`unmount`、`update`）的标准组件。

- **数据传递格式**：前后端通信统一使用 **JSON**，禁止传递复杂的二进制数据（大文件直接传递文件路径字符串）。
